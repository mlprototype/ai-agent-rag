# Project One-Pager: Agentic RAG / Structured Query 基盤

## 1. このシステムは何か

このシステムは、ユーザーの質問内容に応じて **直接応答、非構造化データ検索（RAG）、構造化データ問い合わせ（Structured Query）** を動的に切り替える **Agentic Query / RAG 基盤** です。

単一のRAGパイプラインにすべての質問を流すのではなく、ルーターが質問の性質を判定し、最適な処理経路を選択します。
これにより、社内ドキュメント検索、仕様比較、売上や件数などの集計問い合わせといった、性質の異なる問い合わせを同一の対話インターフェースで扱えます。

## 2. 何を解決するか

従来の単一経路型RAGでは、質問の種類にかかわらず「検索してから答える」流れになりやすく、不要な検索コストやレイテンシ、適切でない回答方式の選択が発生しやすいという課題があります。

本システムは、主に以下の課題を解決します。

*   **無条件検索の回避**
    挨拶や短い定型応答、検索コストに見合わない単純な問い合わせに対して、不要な Retrieval を行わないようにします。
*   **データ種類に応じた経路の使い分け**
    非構造化文書に向く質問はRAGへ、集計・件数・ランキングなどの構造化問い合わせは Structured Query へ振り分け、質問に合った処理方式を選択します。
*   **品質 / レイテンシ / 縮退の制御**
    検索結果が不十分な場合や、タイムアウト・予算不足が発生した場合でも、段階的なフォールバックにより応答継続性を確保します。
*   **企業利用を意識した安全性の確保**
    情報不足時に推測で補完せず、Strict RAG Policy や安全な Structured Query 実行により、誤回答や危険な実行を抑制します。

## 3. コア価値

*   **Routing**
    質問の性質に応じて、直接応答、RAG、Structured Query、比較専用経路などへ動的に振り分けます。
*   **Retrieval**
    非構造化文書に対して、Hybrid Search とコンテキスト抽出を用いて、根拠付き回答を支える検索処理を行います。
*   **Structured Query**
    集計・件数・ランキングなどの構造化問い合わせを、段階的な検証を経た read-only クエリとして安全に実行します。
*   **Budget / Fallback**
    処理時間、品質不足、外部依存の失敗を考慮しながら、段階的に縮退して応答継続性を確保します。
*   **Observability / Evaluation**
    ルーティング結果、レイテンシ、fallback、品質劣化を継続的に測定・分析し、制御品質と検索品質の両方を評価できるようにします。

## 4. 普通のRAGとの違い

本システムは、一般的な「質問 → 検索 → 回答」という直線的なRAGとは異なり、**制御プレーンを持つ問い合わせ基盤** として設計されています。

主な違いは以下の通りです。

*   **常に検索しない**
    ルーターが検索不要と判断した場合は、検索を行わず適切な別経路で処理します。
*   **比較専用の処理経路を持つ**
    単純な検索ではなく、複数対象の比較に適した専用ロジックを持っています。
*   **構造化問い合わせを別経路で処理する**
    テキスト検索だけでなく、売上・件数・ランキングなどの構造化データ問い合わせを別系統で扱います。
*   **Strict Policy / Fallback / Budget Control を備える**
    検索要否、縮退方法、時間予算、品質不足時の挙動まで制御対象に含めています。
*   **評価対象が回答精度だけでない**
    RAG精度だけでなく、ルーティング精度、fallback率、degraded率、レイテンシも含めて評価対象としています。

## 5. 一番重要な設計判断

*   **LangGraphで明示的な状態遷移を採用**
    暗黙的なエージェント挙動に依存せず、状態遷移を明示的に定義することで、制御可能性、予測可能性、可観測性を高めています。
*   **Prompt Runtime は Local Snapshot を正本とする**
    プロンプトをローカルでバージョン管理し、実行時の安定性とレビュー可能性を優先しています。
*   **Hallucination 抑止を優先する**
    情報が不足している場合は推測で補完せず、「十分な根拠がない場合は答え切らない」設計を採用しています。
*   **責務分離により拡張しやすい構成を採用する**
    Retrieval、Structured Query、Ingestion などの責務を分離し、新しいツールやデータソースを追加しやすい構成にしています。
*   **Structured Query は安全性を優先した段階制御にする**
    構造化問い合わせは自由な実行に任せず、Parse → Validate → Build → Execute → Format の段階に分け、read-only 制約のもとで安全に処理します。

## 6. 想定ユースケース

*   社内ドキュメントや仕様書を対象にしたナレッジ検索
*   製品や技術要素の比較問い合わせ
*   売上、在庫、件数、ランキングなどの構造化データ集計
*   情報不足時に無理に答えず、安全に応答する企業向け問い合わせ基盤
*   ルーティングや検索品質の変化を継続的に評価するAIシステム基盤

## 7. 現時点の制約

*   比較経路は主に2対象比較を前提としており、多対象比較には未対応です。
*   Structured Query は安全性を優先した限定的な問い合わせに絞っており、汎用的なSQL Agentではありません。
*   非構造化検索は日本語全文検索の高度な形態素解析までは扱っていません。
*   メモリは永続化されておらず、プロセス再起動で会話履歴は失われます。

## 8. このシステムの位置づけ

このシステムは、単なるRAGアプリケーションではなく、質問の性質・データ種別・品質制約・時間制約に応じて処理経路を制御する、**企業利用志向の Agentic Query 基盤** と位置づけられます。
非構造化知識検索と構造化問い合わせを統合しつつ、回答品質だけでなく制御品質まで評価対象に含める点が中核的な特徴です。
