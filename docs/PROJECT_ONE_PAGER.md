# Project One-Pager: AI Agent RAG 評価ダッシュボード基盤

## 1. このシステムは何か

このシステムは、ユーザーのクエリに応じて適切な処理経路を動的に選択する **Agentic RAG 基盤** です。
主な特徴は以下の通りです：
*   **非構造化データ** は RAG (Retrieval-Augmented Generation) 経路で処理。
*   **構造化データ** は Structured Query (データベースやAPIへの構造化クエリ) 経路で処理。
*   **ルーター (Router)** がクエリの意図を解釈し、最適な経路を自動的に選択します。
*   社内ドキュメント検索、仕様比較、売上集計など、異なる性質の問い合わせを単一の対話インターフェースで扱うことを想定しています。


## 2. 何を解決するか

従来の単一経路のRAGが抱える以下の課題を解決します：
*   **無条件検索の回避**: 挨拶や雑談、検索を必要としないクエリに対して無駄な検索（Retrieval）を行うのを防ぎます。
*   **データ種類に応じた経路の使い分け**: テキスト検索に不向きな集計・比較処理（構造化データ）を適切な処理系に流します。
*   **品質 / レイテンシ / 縮退の制御**: 検索結果が不十分な場合や、タイムアウトが発生した場合に、段階的にフォールバック（縮退動作）を行い、ユーザーへの応答（レイテンシと最低限の品質）を保証します。

## 3. コア価値

*   **Routing**: クエリの性質を高精度に分類し、RAG、Structured Query、QA等の経路へ振り分ける高度なルーター。
*   **Retrieval**: 非構造化データに対する高精度な意味検索と、コンテキストの抽出。
*   **Structured Query**: 構造化データに対して、意図解析結果をもとに安全な read-only クエリへ変換し、集計・検索を実行する仕組み。
*   **Budget / Fallback**: 処理時間（Budget）を管理し、失敗時やタイムアウト時には安全な状態（Fallback）へ移行する堅牢なエラーハンドリング。
*   **Observability / Evaluation**: ルーティング結果、レイテンシ、fallback、劣化状態を計測し、品質回帰や経路ごとの挙動を継続的に分析できる仕組み。

## 4. 普通のRAGとの違い

*   検索要否の制御、安全な構造化問い合わせ、Strict RAG Policy、段階的フォールバックにより、企業利用を意識した堅牢性を重視している。
*   **常に検索しない**: ルーターの判断により、検索が不要な場合は直接回答を生成します。
*   **比較専用経路がある**: 単純な検索ではなく、複数の情報を比較検討するための専用ロジックを持っています。
*   **構造化データを別系統で処理**: テキストチャンクだけでなく、DBなどの構造化データに対するクエリ生成・実行を分離しています。
*   **Strict Policy / Fallback / Budget Control**: ハルシネーションを厳格に防ぐポリシー、タイムアウト時の段階的なフォールバック、そして全体の処理時間を意識したバジェット管理が組み込まれています。

## 5. 一番重要な設計判断

*   **LangGraphで明示的な状態遷移を採用**: 暗黙的なAgentの自律性に頼るのではなく、LangGraphを用いて状態遷移を明示的に定義し、制御可能性と予測可能性を高めています。
*   **Prompt RuntimeはLocal Snapshot正本**: プロンプトはコードベースと密結合させ、バージョン管理可能なローカルスナップショットを正として管理しています。
*   **Hallucination（ハルシネーション）抑止を優先**: 情報が不足している場合は「分からない」と答えることを重視する厳格なポリシーを採用しています。
*   **Tool増設可能な責務分離**: 新しいデータソースや処理ロジックを容易に追加できるよう、ツールごとの責務を明確に分離したモジュラーアーキテクチャを採用しています。
